# AI 위험성평가 도우미 웹앱 개발 계획서

## 1. 프로젝트 개요

### 1.1 목적
건설 현장의 **불시·단발성 작업**에 대한 위험성평가서를 AI가 자동으로 작성해주는 웹 애플리케이션을 개발한다. 사용자가 작업 정보를 입력하면, 회사 보유 위험성평가 DB(엑셀)를 RAG 검색으로 조회하여 회사 표준 양식에 맞춰 자동으로 채워진 엑셀 파일을 다운로드할 수 있게 한다.

### 1.2 핵심 기능 요약
1. 사용자 입력: **작업내용 / 인력 / 장비**
2. RAG 검색: 위험성평가 DB(40개 공종 시트)에서 유사 위험요인·대책 검색
3. AI 생성: 회사 양식에 맞는 위험요인·안전보건추진계획 자동 작성
4. 엑셀 다운로드: 회사 표준 양식의 완성된 위험성평가서

## 2. 입력/출력 명세

### 2.1 입력 (사용자 폼)

| 업체명/공종 | text | O | 예: ㈜한창테크 / 금속 |

### 2.2 출력 (엑셀 파일)
- 회사 표준 양식 `불시,단발성작업(양식)` 그대로 (1행/12행 표)
- 헤더 영역: 현장명, 업체명/공종, 작업기간, 작업인원, 건설기계, 작업 책임자/근로자, 기계/기구 및 위험물질

## 5. RAG 파이프라인 설계

### 5.1 인덱싱(1회 실행, 사전 작업)
1. **소스 파일 변환**: 원본이 구형 `.xls`이므로 LibreOffice headless(`soffice --headless --convert-to xlsx`) 또는 `xlrd==1.2.0`(레거시)로 1회 `.xlsx` 변환 → 이후 `openpyxl`/`pandas`로 안정적 파싱. 변환된 파일은 `data/source/risk_db.xlsx`로 영속화.
2. `pandas.read_excel(sheet_name=None)`로 40개 시트 전체 로드
3. 시트별로 행 단위 정규화:
   ```json
   {
     "id": "sheet=(19)창호 및 유리 작업|row=12",
     "공종": "창호 및 유리 작업",
     "세부작업": "출입구 도어 시공",
     "위험요인": "그라인더 보호덮개 미설치 상태에서 작업 중 ...",
     "재해형태": "비래",
     "안전대책": "날접촉방지 보호덮개 설치 ...",
     "위험등급": "..."
   }
   ```
4. 임베딩 대상 텍스트:
   `f"[공종] {공종} [세부작업] {세부작업} [위험요인] {위험요인} [대책] {안전대책}"`
5. ChromaDB에 메타데이터와 함께 저장 → `data/risk_db/`
6. **재인덱싱 트리거**: 소스 엑셀의 mtime/해시가 변경되면 재인덱싱. CLI 명령(`python -m rag.ingest --force`) 제공.

### 5.2 질의(런타임)
1. 사용자 입력을 쿼리 텍스트로 변환:
   `f"작업내용: {작업내용} / 장비: {장비} / 인원: {인원}"`
2. 1차 검색: **장비 키워드 사전 필터** (그라인더 → 위험기계기구·금속 작업 등 후보 공종 가중). 매핑 사전(`config/equipment_to_trade.yaml`)으로 관리하여 신규 장비 추가 용이.
3. 2차 검색: 임베딩 유사도 top-k(예: k=20) 회수. 필터 결과가 5건 미만이면 필터를 해제하고 전체 인덱스에서 재검색(fallback).
4. 재정렬: LLM 또는 키워드 매칭 점수 기반 reranking → top-8 ~ 12 선정

### 5.3 LLM 합성 (Gemini)
- **모델**: `gemini-2.5-pro` (기본) / `gemini-2.5-flash` (빠른 모드 / 비용 절감)
- **응답 형식**: `responseMimeType: "application/json"` + `responseSchema`로 구조화 출력 강제
- 프롬프트는 다음을 포함:
  - 회사 양식 컬럼 정의 (작업장소/작업내용/위험요인/안전보건추진계획)
  - 사용자 입력
  - RAG 검색 결과 top-k (출처: 공종/세부작업/위험요인/대책)
  - 출력 형식: **JSON 배열**, 각 row = `{작업장소, 작업내용, 위험요인, 안전보건추진계획, 비고}`
  - 규칙: 공통사항(중량물·용접·비산먼지 등) 자동 보강, 입력 장비에 맞는 위험요인 우선
- 검증: JSON 스키마 검증(pydantic) → 실패 시 1회 자기수정 호출 → 그래도 실패 시 검색 결과 원문을 그대로 표 형태로 fallback 반환(사용자에게 "AI 보강 실패, 원본 검색결과 표시" 배지 노출).
- 토큰 예산: 입력 컨텍스트 RAG 결과를 top-k=12, 각 항목 ~300토큰으로 제한(총 ~4k). 출력 max_tokens=4096.

### 5.4 Thinking 수준 (UI 드롭다운)
사용자가 입력 폼 또는 설정에서 **사고(Thinking) 수준**을 선택. Gemini API의 `generationConfig.thinkingConfig.thinkingBudget`로 매핑.

| 옵션 | thinkingBudget | 모델 | 특성 |
|---|---|---|---|
| **빠름 (Fast)** | `0` (비활성화) | `gemini-2.5-flash` | 가장 빠름·저렴, 단순 작업용 |
| **표준 (Balanced)** | `-1` (동적, 기본값) | `gemini-2.5-pro` | 모델이 자동 판단 |
| **신중 (Thorough)** | `8192` 등 고정 큰 값 | `gemini-2.5-pro` | 복잡/다항목 작업 시 권장 |
| **최대 (Max)** | `24576`(또는 모델 상한) | `gemini-2.5-pro` | 고난이도·정합성 우선 |

- API 호출 예 (요지):
  ```json
  {
    "generationConfig": {
      "responseMimeType": "application/json",
      "responseSchema": { ... },
      "thinkingConfig": { "thinkingBudget": -1, "includeThoughts": false }
    }
  }
  ```
- UI에서 선택값을 백엔드 `/api/assess`의 `thinking_level` 파라미터로 전달 → 위 매핑 적용.

## 6. 엑셀 생성 (양식 정합성)

### 6.1 접근법: **템플릿 클론**
- `template/위험성평가서_template.xlsx`를 미리 만들어둠 (양식 PDF를 엑셀로 1회 정밀 재현; 셀 병합·테두리·폰트·로고 포함)
- 런타임에는 템플릿을 복사 후 정해진 셀 좌표에 값만 채움
  - 셀 좌표 매핑은 `template/cell_map.yaml`로 외부화하여 양식 변경 시 코드 수정 없이 대응
    ```yaml
    site_name: "B3"      # 현장명 (실제 좌표는 템플릿 제작 후 확정)
    vendor_trade: "F3"   # 업체명/공종
    period: "H3"         # 작업기간
    body_rows:
      start: "B7"
      end_default: "F18"
      columns: [location, work, hazard, control, note]
    ```
- 12행을 초과하면 행을 자동 삽입(`openpyxl.worksheet.copier`로 서식 유지)
- **검증 테스트**: 생성된 xlsx를 LibreOffice로 PDF 변환 후 원본 양식 PDF와 픽셀 단위 비교(샘플 1건)

### 6.2 다운로드 엔드포인트
- `POST /api/assess` → 평가 결과 JSON 반환 (미리보기용)
- `POST /api/download` → 같은 결과로 xlsx 바이너리 응답
  - `Content-Disposition: attachment; filename=...`

## 8. API 명세 (요약)

### `POST /api/assess`
**Request**
```json
{
  "site_name": "...", "vendor": "...", "trade": "...",
  "period": { "start": "2026-04-28", "end": "2026-04-28" },
  "headcount": 2, "leader": "김재한", "workers": ["김영열"],
  "equipment": ["핸드그라인더", "용접기"],
  "machinery": "해당없음",
  "locations": ["현장출입구", "감리실 외부 출입구"],
  "work_description": "출입구 도어 시공, 몰딩 시공",
  "thinking_level": "balanced"   // "fast" | "balanced" | "thorough" | "max"
}
```
**Response (200)**
```json
{
  "rows": [
    {"location": "현장출입구", "work": "출입구 도어 시공",
     "hazard": "...", "control": "...", "note": ""}
  ],
  "sources": [{"sheet": "(19)창호...", "row_id": "..."}, ...],
  "meta": { "model": "gemini-2.5-pro", "thinking_budget": -1, "fallback_used": false }
}
```
**Error (4xx/5xx)**
```json
{ "error": { "code": "LLM_JSON_INVALID", "message": "...", "trace_id": "..." } }
```
주요 에러 코드: `INPUT_VALIDATION`, `RAG_NO_MATCH`, `LLM_TIMEOUT`, `LLM_JSON_INVALID`, `EXCEL_RENDER_FAILED`.

### `POST /api/download`
- 같은 입력 + (선택) 사용자가 수정한 rows → xlsx 파일
- 응답 헤더: `Content-Type: application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`

## 12. 보안 & 개인정보 처리

### 12.1 개인정보(PII)
- 입력에 포함되는 PII: **작업 책임자/근로자 이름**
- 처리 원칙:

  - 생성된 xlsx는 임시 저장 후 다운로드되며, 서버 디스크에는 24시간 후 자동 삭제

## 13. 테스트 전략

| 계층 | 도구 | 대상 |
|---|---|---|
| 단위 | pytest | RAG retriever, 프롬프트 빌더, 셀 좌표 매퍼 |
| 통합 | pytest + httpx | `/api/assess`, `/api/download` end-to-end |
| 양식 검증 | LibreOffice + 이미지 diff | 생성 xlsx ↔ 원본 양식 PDF |
| LLM 회귀 | 골든 케이스 5~10건 | 입력 → 기대 출력 키워드 포함 여부 |
| 프론트 e2e | Playwright | 입력 → 생성 → 편집 → 다운로드 플로우 |

- LLM 회귀 테스트는 비용 때문에 CI에서는 매 커밋이 아니라 nightly 실행
- 골든 케이스는 `tests/fixtures/` 아래에 입력 JSON과 기대 키워드를 함께 보관
